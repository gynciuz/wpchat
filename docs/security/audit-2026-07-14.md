# ChatAdmin security audit — findings & status (2026-07-14)

Read-only audit of the plugin's attack surface (REST auth, capability gating,
SQLi, upload, SSRF, secrets, XSS, prompt-injection / confirmation integrity).

**Overall posture:** solid on WP-hardening fundamentals — every REST route has a
real `permission_callback`, all custom-table SQL is `$wpdb->prepare()`d, history
is user-scoped (no IDOR), uploads are finfo-sniffed and exclude SVG, LLM
endpoints are hardcoded (no SSRF), git-sync uses `proc_open` with an argv array
(no shell), and the API key never reaches the browser. The material risk is
architectural: the "confirm before mutating" gate is enforced by trusting the
LLM.

### 1 — Confirmation-flag bypass (HIGH) — FIXED
The gate short-circuited on `$args['_confirmed']`, an internal flag the
direct-action REST routes set on a real click. Tool `input` from the model was
passed through unmodified, so a prompt-injection payload could emit
`update_order_status({..., _confirmed: true})` and skip confirmation.
**Fix:** strip every `_`-prefixed key from model-supplied tool input in
`BaseLLMProvider::run_with_tools` before dispatch. Direct-action routes bypass
that loop, so they still work.

### 2 — Model-assertable confirmation (MEDIUM) — FIXED
Every write gate validated only that the `confirmation` string was a whitelisted
affirmative — which the model supplies. Nothing server-side proved a `preview_*`
ran first or that a human consented.
**Fix (approach B):** a server-side `PendingConfirmation` record (per-conversation
transient) is minted by `preview_content_change` / the first `needs_confirmation`,
stamped with the current user-turn index (count of user messages). The matching
apply proceeds only if it supplies a whitelisted phrase **and** consumes a pending
record for that exact target minted in a **strictly earlier** user turn. Prompt-
injected content lives in tool results (not genuine user messages) and cannot
create a new user turn, so it can't satisfy the gate. Direct-action REST clicks
(`_confirmed`) and direct/programmatic callers (no request context) keep their
existing contracts. Tests: `PendingConfirmationTest` (5) +
`ConfirmationTurnGuardTest` (2, incl. the injection case that writes on the old
code and is blocked on the new). Full design + the rejected token approach:
`docs/superpowers/specs/2026-07-14-confirmation-token-design.md`.

### 3 — Onboarding privilege gap (MED/LOW) — FIXED
`/onboarding/api-key|model|provider|llm-provider` were gated at
`manage_woocommerce || edit_shop_orders`; the Settings page reserves the key for
`manage_options`. A shop-manager could set/replace the site-wide key, model, or
provider (billing/DoS/policy vector; key never disclosed). **Fix:** gate those
four routes with `check_admin` (`manage_options`); `/status` stays at the looser
cap. Regression test: shop_manager → 403, injected key not persisted.

### 4 — Inline-JSON `</script>` breakout (LOW) — FIXED
`wp_json_encode($boot)` doesn't set `JSON_HEX_TAG`, so a `</script>` in a site or
user name emitted into the inline `<script>` could break out (self-XSS / needs a
malicious admin). **Fix:** `JSON_HEX_TAG|AMP|APOS|QUOT` on the frontend boot,
admin boot, and support-report inline JSON.

### 5 — Data-sharing disclosure (LOW/INFO) — FIXED
"Report a problem" ships recent conversation (can include customer PII) + the
reporter's login/email to the developer's collector; error telemetry defaults on.
The Settings note only mentioned the AI provider. **Fix:** the note now discloses
both developer channels and how to turn error reporting off. (Making telemetry
opt-in and stripping any PII from telemetry `message` fields is a further
hardening step for the wp.org submission.)

## Verified clean (no action)
REST permission callbacks (8/8 real, none `__return_true`); REST cookie-nonce
CSRF protection on the direct-action routes; capability re-check at dispatch
(`check_kind_access`, `seo_setting` → `manage_options`); all `class-history.php`
SQL prepared; upload MIME allowlist excludes SVG; hardcoded LLM endpoints;
git-sync argv `proc_open` + path validation; API key write-only + masked;
admin/diagnostics output `esc_*`'d; `ABSPATH` guards everywhere; per-user rate
limiting; no `eval`/`shell_exec`/untrusted `unserialize`.
